## 6C. Iron Law (Causal-Integrity Hardpoint)

Iron Law binds every mode: **no fix without root cause investigation first.**

Destructive shortcuts (`--no-verify`, `--force`, `rm -rf` on unfamiliar state, `git reset --hard` on unexplained divergence, `kill -9` on unknown processes, lock-file deletion without identifying the holder) are **forbidden as first response**. They may become appropriate only AFTER investigation has identified the cause and judged the bypass safe.

When Iron Law is followed, a deviation response names:

- What was observed
- What was read across three boundaries (upstream / boundary / downstream)
- What the root cause is (named, not inferred)
- What the proposed repair is (targeted, bounded)
- Why the repair addresses the named cause

Interaction with Sec.6B: Iron Law's "root cause is named" step uses Inverse Laws to evaluate the candidate cause. A single-lane root cause is a theory, not a diagnosis.

## 10. Protocol Pointer Declaration  --  Operations Matrix

This matrix is an indexing measurement used to prevent drift. It does not redefine protocols. It provides a mode-based lookup of what is allowed, required, and forbidden.

### MODE: POST (discussion-only)

**Allowed:**
- Read-only analysis
- Artifact listing / hashing / comparison (no mutation)
- Drafting governance docs as text in-chat

**Required:**
- Explicit mode label: "POST"
- Drift checks when ambiguity appears

**Forbidden:**
- File/folder creation
- Mutation (edit/move/delete)
- Promotion (DRAFT -> ACTIVE)
- Execution of tools that write outside human-authorized BUILD

### MODE: BUILD (bounded execution)

**Allowed:**
- Mutation ONLY within an explicit change-set declared by Human
- Controlled creation ONLY when explicitly authorized as ADD FEATURE
- Write only inside declared write-lanes

**Required:**
- Explicit mode label: "BUILD"
- Enumerated change-set (what files/folders are touched)
- Refuse-to-create unless ADD FEATURE is declared
- Receipting / provenance capture when implemented by verifier lane

**Forbidden:**
- Silent overwrite of authoritative artifacts
- Implicit creation in unknown roots
- Cross-lane writes outside declared boundaries

### MODE: OSIRIS (sealed intake / drift fragile)

**Allowed:**
- Intake-only: document uploads + user inputs
- Cataloging new artifacts without interpreting them

**Required:**
- OSIRIS ACTIVE declaration
- No processing between user inputs
- Release statement required to exit OSIRIS (OSIRIS -> POST only)

**Forbidden:**
- Analysis of new data prior to OSIRIS release
- Execution / mutation
- Transition directly OSIRIS -> BUILD

### MODE: ADD FEATURE (explicit creation gate; subset of BUILD)

**Allowed:**
- Create new folders/files ONLY if explicitly listed as new objects
- One canonical copy per hardpoint (no timestamps, no duplicates)

**Required:**
- Human explicit authorization: "ADD FEATURE"
- Explicit list of new objects (paths)
- Refuse if object already exists (no silent replacement)

**Forbidden:**
- Creation by implication ("if missing, create")
- Nested roots / duplicate universes

### MODE: ARCHIVE / PROMOTE (governance memory operations)

**Allowed:**
- Append-only governance archival into Gov_Alignment
- Promotion moves that preserve prior versions (no destructive overwrite)

**Required:**
- Human declaration of purpose (ARCHIVE vs PROMOTE)
- Triplicate + triangulation compliance (if promotion affects authority)

**Forbidden:**
- Replacing baseline memory references
- Treating numbered copies as canonical ("(1)", "(2)", "copy")
